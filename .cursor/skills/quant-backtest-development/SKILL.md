# Quant Backtest Development

## 开发原则

1. 先确认需求、现有实现和成功标准，再修改代码；有实质歧义时明确说明假设。
2. 采用能满足需求的最小改动，不添加未要求的抽象、配置或邻近重构。
3. 保持现有分层：数据访问、因子、执行、回测、绩效、绘图各自负责本层逻辑；CLI 只解析参数并编排。
4. 每一处改动都应能追溯到需求。只清理由本次改动造成的无用代码。
5. 默认使用简体中文沟通、写项目文档和提交信息；代码标识符遵循现有英文命名。

## 标准工作流

1. 阅读相关模块、测试、`README.md` 和 `dev_plan.md`，不要仅凭文件名推断行为。
2. 为新行为列出可验证条件。修复缺陷时，优先添加能复现问题的测试。
3. 先运行相关测试，实施局部修改，再运行相关测试和完整质量门禁。
4. 依赖统一写入 `pyproject.toml`；需要更新可复现环境时同步维护 `environment.yml`。
5. 用户可见的接口、命令或行为变化应更新 `README.md`；完成一个开发阶段时更新 `DEVELOPMENT_LOG.md`。
6. 除非用户明确要求修改计划，不改写 `dev_plan.md`。

## 领域不变量

### 数据层

- 标准行情字段为 `code/date/open/high/low/close/volume/adj_factor`，日期按交易日处理。
- `(code, date)` 必须唯一；MongoDB 写入采用唯一索引和幂等 upsert。
- 不前向填充 OHLC。缺失 OHLC 的记录应拒绝或删除；缺失成交量记为 0，表示不可交易。
- `adj_factor` 可按既有规则有限前向填充；复权价格必须显式选择，不能静默替换原始价格。
- A 股成交量统一为“股”，不是“手”；证券代码必须经过现有市场规范化逻辑。
- 股票池快照要维护生效区间，更新成分时正确关闭旧成员，避免历史穿越。
- 外部数据源必须设置超时并隔离单标的失败；测试不得访问实时网络。

### 因子与回测

- 禁止未来数据泄漏。因子信号与成交至少按既有 T/T+1 规则错位，时序计算保留必要的 `shift(1)`。
- 因子输入要拒绝重复 `(code, date)`；窗口按有效行情行数计算。
- 截面处理保持 MAD 去极值、z-score 标准化及零离散截面的既有定义。
- 调仓选择、权重和收益计算不可混用同日未知信息。
- 修改执行模型时同时验证现金、市场差异化佣金、卖出税、滑点、停牌/零量和成交量参与率。
- 零成本兼容路径与带摩擦执行路径都应保持可测试。

### 绘图与 Notebook

- `PriceChartEngine` 只读行情；折线图默认原始 `close`。
- K 线聚合保持 OHLC 的 first/max/min/last 和成交量求和，标题标明实际频率。
- 绘图测试使用 Matplotlib `Agg` 后端，不依赖桌面环境。
- Notebook 使用代码参数，避免引入非必要 widgets；提交前清理大体积临时输出。

## 测试规范

- 单元测试使用固定、最小且能表达边界的数据。
- MongoDB 仓储测试使用 `mongomock`，不得修改本地 `quant_db`。
- 新增功能至少覆盖正常路径、关键边界和明确错误；不要只断言“没有抛异常”。
- 浮点结果使用合适容差；日期、代码、排序和 DataFrame 列应显式断言。
- 外部提供方通过 mock 响应测试解析、单位转换、特殊代码和异常处理。

先运行相关测试，完成后执行完整门禁：

```bash
conda run -n quant pytest
conda run -n quant python -m compileall -q quant_backtest tests
```

只有在任务涉及真实数据链路且本地服务可用时，才补充只读或明确范围的端到端验证。自动化测试必须可离线重复运行。

## Git 与交付

- 未经用户明确要求，不提交、不推送、不创建 PR。
- 提交前检查状态和完整 diff，不提交 `.vscode/`、缓存、密钥、数据库文件或生成图片。
- 提交信息说明改动目的；推送后确认本地分支与远端同步。
- 最终说明改动文件、验证结果、已知限制，以及用户真正需要执行的下一条命令。
